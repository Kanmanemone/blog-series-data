# Concept: 티스토리 시리즈 목차 JSON 유지보수 자동화

- **Slug**: blog-sitemap-sync
- **Created**: 2026-07-21
- **Recommended option**: Option B — lastmod 기반 부분 자동화 + 사람 검토(PR)

## Options

### Option A — 그대로 유지 (수동 프로세스 개선만)

- **Sketch**: 자동화 프로그램을 만들지 않고, 현재처럼 관리자가 새 게시글이 올라올 때 sitemap을 열어 확인하고 직접 `*_series.json`을 갱신한다. 다만 매번 sitemap 전체를 눈으로 훑는 부담을 줄이기 위해, "마지막으로 확인한 시점 이후 `lastmod`가 바뀐 게시글만 보여주는" 간단한 조회용 스크립트(로컬 실행, 저장소에 자동 반영 없음) 정도만 추가한다.
- **Appetite**: small (하루 이내)
- **Trade-offs**: 구현·유지보수 부담이 거의 없고 실수 위험도 낮다. 하지만 문제 정의에서 지적한 "반복적 수동 작업" 자체는 해소되지 않으며, 게시글이 늘어날수록 결국 다시 부담이 커진다. GitHub Actions로 실행한다는 원래 요청과도 맞지 않는다.
- **Rabbit holes**: 없음 — 범위가 매우 작아 확장 위험도 작다. 다만 "결국 자동화가 필요해서 다시 이 작업을 반복하게 되는" 기회비용이 있다.

### Option B — lastmod 기반 부분 자동화 + 사람 검토(PR)

- **Sketch**: GitHub Actions가 주기적으로(cron) 또는 수동 실행(workflow_dispatch)으로 sitemap.xml을 가져와, 저장소에 커밋된 작은 상태 파일(마지막 동기화 시점/마지막으로 처리한 `lastmod`)과 비교해 그 이후 갱신된 게시글만 골라낸다. 골라낸 게시글에 대해서만 제목을 확인하고, 이미 알고 있는 시리즈 이름/패턴과 매칭이 명확한 경우에는 해당 `*_series.json`의 `items`에 자동으로 추가하는 변경을 PR로 올린다. 매칭이 애매하거나 새로운 시리즈로 보이는 경우에는 자동 반영하지 않고 PR 설명이나 이슈에 "확인 필요" 항목으로만 표시해 사람이 최종 판단한다. 상태 파일은 PR이 머지될 때만 갱신되어, 다음 실행의 "마지막 동기화 시점"이 된다.
- **Appetite**: medium (1~2주)
- **Trade-offs**: 매번 sitemap 전체를 다시 확인하지 않는다는 문제 정의의 핵심 목표(정량적 성공 지표)를 달성하면서도, 시리즈 분류처럼 아직 기준이 불확실한 부분은 사람이 최종 확인하므로 오분류 위험을 낮춘다. 대신 완전 무인 자동화는 아니어서 PR을 확인·머지하는 사람의 개입이 여전히 필요하다.
- **Rabbit holes**: (1) 상태 파일을 어디에 어떤 형식으로 저장·갱신할지, PR이 머지되기 전에 다시 워크플로우가 실행되는 경쟁 상황을 어떻게 다룰지. (2) "매칭이 명확한 경우"와 "애매한 경우"를 가르는 기준을 정의하는 일 자체가 예상보다 커질 수 있음. (3) 게시글 삭제/제목 수정 같은 Update/Delete 케이스를 어디까지 이 흐름에 포함할지 범위가 흐려질 수 있음.

### Option C — 완전 자동 커밋(사람 검토 없음)

- **Sketch**: Option B와 같은 lastmod 기반 필터링을 쓰되, 시리즈 매칭에 확신이 없는 경우까지 포함해 모든 변경을 자동으로 커밋해 바로 `main`에 반영한다. 새로운 시리즈로 보이는 게시글 묶음이 발견되면 새 `*_series.json` 파일도 자동으로 생성한다.
- **Appetite**: large (수 주 이상, 특히 분류 로직을 신뢰할 수 있는 수준까지 다듬는 데)
- **Trade-offs**: 사람 개입이 전혀 없어 완전한 "무인 자동화"를 달성하지만, 문제 정의의 열린 질문(시리즈 판별 기준, lastmod의 신뢰도, 삭제 처리)이 모두 해결되지 않은 상태에서는 잘못된 커밋이 그대로 `main`에 들어가 저장소 데이터를 조용히 망가뜨릴 위험이 크다.
- **Rabbit holes**: 시리즈 자동 판별 로직의 정확도를 사람 개입 없이 신뢰할 수 있는 수준까지 끌어올리는 작업, 새 시리즈 파일을 자동 생성할 때 이름/스키마를 결정하는 로직, 잘못된 자동 커밋을 되돌리는 절차 마련까지 범위가 계속 커질 수 있음.

## Recommendation

**Option B**를 추천한다. 문제 정의의 핵심 목표인 "매번 sitemap 전체를 재확인하지 않는다"(정량적 성공 지표)를 lastmod 기반 필터링으로 달성하면서도, 아직 열려 있는 질문들(시리즈 판별 기준, lastmod 신뢰도, 삭제 감지)이 명확해지기 전까지는 사람이 최종 반영 여부를 확인하는 안전장치(PR)를 두는 편이 데이터 손상 위험을 줄인다. Option A는 자동화라는 원래 요청을 충족하지 못하고, Option C는 문제 정의에 남아 있는 불확실성을 그대로 안은 채 완전 자동 커밋으로 넘어가 위험이 크다.

## Out of Scope (for the recommended option)

- 시리즈 판별에 쓰일 구체적인 알고리즘/휴리스틱 설계 (스펙 단계에서 정의)
- 상태 파일의 정확한 포맷과 저장 위치, 워크플로우의 세부 구현 (계획 단계에서 정의)
- 완전 무인 자동 커밋 (Option C의 영역이며, Option B가 검증된 이후에 재검토 대상)
- 기존 `*_series.json` 스키마 자체의 변경
- 게시글 삭제/제목 수정에 대한 완전한 자동 처리 (최초 버전은 신규 게시글 추가 위주로 시작하고, 삭제·수정 처리는 이후 확장으로 남겨둘 수 있음)

## Assumptions to Validate

- sitemap.xml의 `lastmod` 값이 게시글의 실질적인 변경(신규 발행 포함)을 놓치지 않고 반영한다는 가정
- 티스토리 sitemap.xml에서 게시글 제목이나 시리즈 판별에 쓸 수 있는 정보(제목, URL 패턴 등)를 별도 로그인 없이 얻을 수 있다는 가정
- PR 기반 검토 흐름이 실제 관리자의 운영 방식(얼마나 자주 PR을 확인·머지할 수 있는지)과 맞는다는 가정
- 저장소에 상태 파일을 하나 추가하는 것이 기존 헌법(스키마 일관성 원칙)과 충돌하지 않는다는 가정
